# Мой ежедневник-органайзер  
## Философия и управленческая модель

## 1. Назначение системы

Система предназначена для личного управления задачами, встречами и мыслями
через минимальное количество действий и максимальную фиксацию фактов.

Цель — не «вести список задач», а:
- не терять входящий поток,
- контролировать сроки и нагрузку,
- принимать управленческие решения о времени,
- видеть картину целиком: сегодня / неделя / проекты / крупные задачи.

Система строится как долговременный инструмент, а не MVP «на попробовать».

---

## 2. Принципиальные положения (CORE)

### 2.1 Источник истины
- Единственный источник истины — **база данных системы**.
- Telegram, Google Calendar, Excel — это интерфейсы и проекции.
- Если данные не зафиксированы в базе — их не существует.

### 2.2 Telegram — основной интерфейс
- Telegram используется как быстрый фронтенд:
  - голос,
  - текст,
  - команды,
  - кнопки.
- Пользователь не обязан помнить команды.
- Свободная речь допустима и поощряется.

### 2.3 Inbox как обязательный буфер
- Весь входящий поток сначала попадает в Inbox.
- Inbox — страховка от потери задач.
- Разбор Inbox — управленческое действие, а не автоматизм.

### 2.4 Управление временем важнее управления задачами
- Задача без времени — это потенциальный хаос.
- Встречи и задачи с временем имеют приоритет над абстрактными списками.
- Перегруз дня считается ошибкой планирования.

### 2.5 Интеллект — подключаемый, но не обязательный
- Система обязана работать **без LLM**.
- Интеллект:
  - усиливает,
  - анализирует,
  - предлагает,
  - но не блокирует работу.
- При отсутствии модели система деградирует gracefully.

---

## 3. Режимы работы

### 3.1 Быстрый режим
- Всё принимается без анализа.
- Минимум вопросов.
- Максимум фиксации.

### 3.2 Управленческий режим
- Разбор Inbox.
- Планирование дня/недели.
- Переносы, отмены, приоритизация.
- Осознанные решения.

---

## 4. Структура верхнего уровня (зафиксировано)

Проекты первого уровня (создаются автоматически):

1. Inbox (системный)
2. Личные
3. Инфомед
4. Проекты
5. Подумать
6. Обо всём

Расширение возможно без изменения философии.

---

## 5. Что принципиально НЕ делаем

- Не делаем командную систему.
- Не делаем социальные функции.
- Не строим систему вокруг LLM.
- Не усложняем UI ради UI.
- Не храним «планы в голове».

## Stage 0 — факты и принципы

### Stage 0 — зафиксированные принципы (по факту E2E)

- **SQLite/Organizer DB — единственный источник истины.**
  - Telegram — транспорт.
  - Google Calendar — асинхронная проекция.
- **Данные из SQLite приходят как `sqlite3.Row`.**
  - Бизнес-логика работает через `row["field"]` **или** делает `dict(row)` один раз на входе.
- Ответ бота в Telegram **не равен** гарантии появления события в календаре.
- Любая автоматизация делается по схеме: **сначала записали в БД → затем пытаемся выполнить действие**.
- Сбой ASR/Calendar API/сети **не должен приводить** к потере записи: максимум — остаётся `inbox/active` без календарного `event_id`.
